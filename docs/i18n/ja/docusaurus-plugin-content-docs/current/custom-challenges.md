# カスタムのスパム対策の課題

Bitsocial は、単一の普遍的なスパム防御を想定していません。各コミュニティは、有効な投稿や返信に必要なものを決定でき、そのチャレンジは、オペレータの要望に応じて軽量または独自の意見にすることができます。

コミュニティ アドレスからチャレンジゲート型投稿までの完全な平易な英語の流れについては、以下を参照してください。 [Bitsocial プロトコルの完全な素人説明](./layman-protocol-explanation.md).

## チャレンジとは何であるか

- キャプチャ
- アカウント年齢または評判のチェック
- SMS認証
- 少額の支払い
- トークンまたはNFTの要件
- IPベースルール
- ホワイトリスト
- コードで表現できるその他のポリシー

## これが重要な理由

スパム耐性は、プロトコル全体の管理機関に委託されるのではなく、コミュニティ内でローカルに維持されます。コミュニティをホストするノードは、チャレンジをピアツーピアで通信できます。つまり、ネットワーク全体を 1 つのポリシーに強制することなく、コミュニティが防御を適応させることができます。

## これによる変更点

集中型プラットフォームでは、通常、スパム対策ポリシーが製品ポリシーになります。そうなると、1 つの企業が、何が有効なアカウント、有効な投稿、または許容可能なリーチとしてカウントされるかを決定します。

Bitsocial はその結合を打ち破ります。

- プロトコルは中立を保ちます
- コミュニティがチャレンジを選択します
- アプリがそのチャレンジを提示または抽象化する方法を決定します

これは、1 つのグローバルなデフォルトではなく、多くの種類のソーシャル プロダクトをホストするネットワークに適しています。
