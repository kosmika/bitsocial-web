# Bitsocial の主力アプリ

主力の Bitsocial アプリは、ネットワーク向けに提案されている最初のプロファイルベースのクライアントです。最も単純な比較は、X/Twitter のようにデフォルトでおなじみですが、プラットフォーム層が壊れていることです。

基盤となるサービスを置き換え可能にしながら、プロフィール、フォロー、返信、フィード、コミュニティ、通知、リアルタイムの公開会話を追加します。 5chan は、匿名コミュニティが機能することを証明しました。 Seedit は粘り強い議論に向けて動きます。主力アプリは、1 つの企業がグラフの所有者になることなく、これらのネットワーク効果を主流のソーシャル フィードに組み込むことになります。

このページは製品の方向性を説明するものであり、ロック解除の仕様ではありません。正確なインターフェイス、デフォルトのフィード、広告モデル、AI 機能、RPC マーケットプレイスは、プロトコルや初期のアプリが成熟するにつれて変更される可能性があります。

## 何を証明すべきか

アプリは、プロフィールベースのソーシャル ネットワークが保管プラットフォームになることを回避できることを証明する必要があります。

- ユーザーはアイデンティティとプロファイルを所有できる
- コミュニティとプロファイル ノードはピアツーピアを維持できる
- コミュニティは Bitsocial クライアント全体にネットワーク効果をもたらすことができます
- RPC プロバイダーは、管理を行わずにアプリを便利にすることができます
- フィード アルゴリズムはプラットフォーム法の代わりにオプションのサービスになる可能性がある
- 他のクライアントは引き続き同じネットワークをめぐって競合することができます

重要なのは、唯一の Bitsocial クライアントを作成することではありません。重要なのは、プロトコルがどこまで拡張できるかを示す最初の広範なクライアントを作成することです。

## デフォルトで馴染みがあり、デザインによって置き換え可能

デフォルトのエクスペリエンスは、主流の公開会話アプリと競合できるものである必要があります。高速なホーム フィード、フォロー、返信、再投稿スタイルの配信、コミュニティ、通知、検索、すぐに機能するランク付けされた For You ビューなどです。

Bitsocial Forge は、最初のデフォルト RPC およびフィード サービスを実行できます。このデフォルトには、ランク付けされたフィードと広告を含めることができるため、メインストリーム ユーザーにスタック全体を自分で組み立てるように依頼するのではなく、初日からアプリが完成したと感じることができます。

違いは、デフォルトが刑務所になってはいけないということです。ユーザーは、RPC、フィード、インスタンス、ランキング システム、広告、ディスカバリー プロバイダーを切り替えたり、ランキングを完全に削除したりできる必要があります。すべての主要なサービスを置き換え可能にしながら、最初の起動時にアプリの意見を決めることができます。

これにより、アプリは従来のプラットフォームよりもカスタマイズしやすくなります。あるユーザーは、広告付きのデフォルトのランク付けフィードを保持する場合があります。別の場合は、ランキングのない時系列フィードを使用する場合もあります。プライバシーを重視した RPC、コミュニティが運営するディスカバリー サービス、広告なしの有料フィード、または特定のサブカルチャー向けに構築されたニッチなアルゴリズムを選択する人もいます。

## クライアントを超えたコミュニティ

コミュニティは、1 つのアプリ内の孤立したグループよりもはるかに重要である必要があります。

X/Twitter では、コミュニティは X 内に限定されます。コミュニティは便利ですが、1 つのプラットフォーム、1 つのアカウント システム、1 つの推奨スタック、および 1 つの製品サーフェスの制限を受け継ぎます。

Bitsocial コミュニティは、さまざまなクライアントを通じて作成、ホスト、検出、使用できます。つまり、フラッグシップ アプリでは、フラッグシップ アプリ内で開始したユーザーだけでなく、より広範な Bitsocial ネットワークからのコミュニティや投稿を表示できるということです。コミュニティには、イメージボード クライアント、Reddit スタイルのディスカッション クライアント、ニッチ フォーラム クライアント、モバイル アプリ、および主力アプリからのアクティビティが同時に存在する可能性があります。

これがネットワーク効果の中核的な利点です。1 つのクライアントが主流のユーザーにとって親しみを感じながらも、多くのクライアント、コミュニティ ノード、RPC プロバイダー、および独立したサービスから価値を引き出すことができます。

## オプションのフィードアルゴリズム

主力アプリは、全員に 1 つのグローバル ランキング システムを強制すべきではありません。

フィードアルゴリズムはオプトインである必要があります。ユーザーは、マーケットプレイスからアルゴリズムを選択することも、プロバイダーを切り替えることも、企業のアルゴリズムを使用することも、匿名のオペレーターが実行するアルゴリズムを使用することも、コミュニティによって構築されたアルゴリズムを使用することも、個人的なアルゴリズムを実行することも、あるいはアルゴリズムをまったく使用しないこともできます。

パブリック RPC プロバイダーは、これらのサービスが競合するのが自然な場所です。コンテンツのインデックス付け、ランク付け、推奨はできますが、ユーザーやプロフィールを所有すべきではありません。

これらのサービスは、アプリ自体の形状でも競合する可能性があります。 1 つの RPC は、広告を含むランク付けされたフィードを提供する場合があります。また、ランク付けされていない時系列フィードを提供する場合もあります。プライバシー、翻訳、モデレーション、コミュニティ発見、またはニッチなソーシャル グラフを専門とする人もいます。

経済性がうまくいけば、RPC 支援のフィード サービスは、自動翻訳、要約、ボット支援による返信、検索回答、モデレーション支援、コミュニティ ノート スタイルのコンテキストなど、主流プラットフォームがフィードに導入しようとしているものと同様の AI 機能を追加できる可能性があります。

これらの機能は、プロトコル要件ではなく、サービスの選択である必要があります。デフォルトの RPC は、より豊富なフィードを提供することで競合できますが、ユーザーと競合するクライアントは、よりシンプル、プライベート、時系列、広告なし、またはコミュニティ固有の代替を選択できる必要があります。

## 非保管 RPC

各ユーザーは、自分の ID やプロファイルに対する所有権を RPC プロバイダーに与えることなく、RPC を通じて完全なピアツーピア ノードとして参加できる必要があります。

ほとんどのユーザーはサーバーを実行することから始めないため、ホストされたパスは重要です。終了パスも同様に重要です。ユーザーはいつでも、Raspberry Pi を含む低スペックのハードウェア上の自分のプロファイル ノードに移動できる必要があります。

それが利便性と保管性の違いです。

## なぜ万能アプリになり得るのか

Bitsocial Chain がアプリに永続的なネーミング、支払い、投げ銭、賞品、その他の経済的レールを提供すれば、主力アプリは単なるフィード クライアント以上のものになる可能性があります。

重要な制約は、アプリがネットワークの新しい所有者になってはいけないということです。それは、競合するアプリ、競合する RPC、競合するフィード アルゴリズム、および自己ホスト型プロファイル ノードのための余地を残しながらも、大規模なクライアント、おそらく最も人気のあるクライアントになる可能性があります。
