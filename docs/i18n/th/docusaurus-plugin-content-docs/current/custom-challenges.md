# ความท้าทายในการต่อต้านสแปมแบบกำหนดเอง

Bitsocial ไม่มีการป้องกันสแปมแบบสากลเพียงครั้งเดียว แต่ละชุมชนสามารถตัดสินใจได้ว่าโพสต์หรือการตอบกลับที่ถูกต้องนั้นต้องการอะไร และความท้าทายนั้นอาจมีขนาดเล็กหรือให้ความคิดเห็นตามที่ผู้ดำเนินการต้องการ

หากต้องการทราบขั้นตอนภาษาอังกฤษธรรมดาฉบับเต็มตั้งแต่ที่อยู่ของชุมชนไปจนถึงการโพสต์แบบมีรั้วรอบขอบชิด โปรดอ่าน [คำอธิบายทั่วไปที่สมบูรณ์ของโปรโตคอล Bitsocial](./layman-protocol-explanation.md).

## เป็นสิ่งที่ท้าทาย

- แคปช่า
- การตรวจสอบอายุบัญชีหรือชื่อเสียง
- การยืนยันทาง SMS
- การชำระเงินเล็กน้อย
- ข้อกำหนดโทเค็นหรือ NFT
- กฎเกณฑ์ตาม IP
- รายการที่อนุญาต
- นโยบายอื่นใดที่สามารถแสดงเป็นโค้ดได้

## ทำไมเรื่องนี้ถึงสำคัญ

การต่อต้านสแปมจะคงอยู่เฉพาะในชุมชน แทนที่จะถูกจ้างจากภายนอกไปยังหน่วยงานกำกับดูแลทั่วทั้งโปรโตคอล โหนดที่โฮสต์ชุมชนสามารถสื่อสารความท้าทายแบบเพียร์ทูเพียร์ ซึ่งหมายความว่าชุมชนสามารถปรับการป้องกันของตนได้โดยไม่ต้องบังคับเครือข่ายทั้งหมดให้เป็นนโยบายเดียว

## สิ่งที่เปลี่ยนแปลงนี้

บนแพลตฟอร์มแบบรวมศูนย์ นโยบายป้องกันสแปมมักจะกลายเป็นนโยบายผลิตภัณฑ์ เมื่อสิ่งนั้นเกิดขึ้น บริษัทหนึ่งจะตัดสินใจว่าอะไรถือเป็นบัญชีที่ถูกต้อง โพสต์ที่ถูกต้อง หรือการเข้าถึงที่ยอมรับได้

Bitsocial ทำลายการเชื่อมต่อนั้น:

- โปรโตคอลยังคงเป็นกลาง
- ชุมชนเลือกความท้าทาย
- แอพจะตัดสินใจว่าจะนำเสนอหรือสรุปความท้าทายนั้นอย่างไร

นี่เป็นวิธีที่เหมาะสมกว่าสำหรับเครือข่ายที่มีไว้เพื่อโฮสต์ผลิตภัณฑ์โซเชียลหลายประเภท แทนที่จะเป็นค่าเริ่มต้นเดียวทั่วโลก
